Arbitrary Plain Text Within XML Using Entity Substitution and CDATA Sections

Background

XML is a format for representing data as a nested hierarchy of elements and text. At the top level, there is a root element, and within the root element or any child element, there can be elements or text, and the text may also contain elements.

The start of an element is represented by an opening tag, which is an element name surrounded by angle brackets (less-than and greater-than characters) and the end of an element is a closing tag, which is like the opening tag except that it includes a slash following the left angle-bracket character.

Example: <SomeElement>Some text.</SomeElement>

Additionally, an element with no content may be optionally represented as a self-closing element tag. Such a tag has a slash directly preceding the right angle-bracket.

Example: <SomeSimpleElement/>

Of course, you might want the text content to contain characters with special meaning to XML such as the less-than character “<”. XML provides 2 choices of mechanism for dealing with that situation: XML entities and CDATA sections.

XML Entities

In XML, an entity is designated by a name preceded by an ampersand and followed by a semicolon. XML uses specific entites as substitites for characters in text content that would otherwise have special meaning in the XML markup. In particular, “&lt;” represents a less-than character “<” and “&gt;” represents a greater-than character “>”. Also, since the ampersand represents the start of an entity, the ampersand character itself must be represented as an entity, which is “&amp;”.

CDATA Sections

In addition to the use of entities, XML supports what are known as a CDATA sections. Each CDATA secion begins with “<![CDATA[” and ends with “]]>”. All of the text within the CDATA section represents itself directly, and no entities need to be used. In fact, no entities may be used because anything in the form of an entity will be simply treated as that exact sequence of characters. As an example, when contained in a CDATA section, “&lt;” represents the text “&lt;”, not the text “<”.

The only text that cannot be contained in a CDATA section is the character sequence of “]]>” since that is what is used to indicate the end of the CDATA section.

Example
<![CDATA[Example XML: <SelfClosingElementTag/>]]>
Represents Text
Example XML: <SelfClosingElementTag/>

Since entity substitution is available, CDATA is technically unnecessary. CDATA blocks are convenient, however, for use in XML documents that are written or edited directly by humans using a text editors.

Processing Directives and Declarative Statements

In the description of CDATA sections above, you might have noticed that the left angle-bracket designated the start of something other than an element, so obviously, XML uses the angle brackets for some things besides elements.

XML actually uses angle-brackets to represent at least 2 categories of thing that are not elements. Those are processing directives (surrounded by “<?” and “?>”) and declarative statements (surrounded by “<!” and “>”). A CDATA section is a kind of declarative statement. A common example of a processing directive is the XML prologue that may appear at the start of an XML document. Common examples of declarative statements are comment blocks and CDATA sections.

XML Prologue Example
<?xml version="1.0" encoding="UTF-8" standalone="yes"?>
Comment Block Example
<!-- This is a comment -->

On Generation and Parsing of XML Documents

The full specification of XML is big and complicated. There are several commonly used, standards-compliant code libraries for generating and parsing XML and you should nearly always use one of those and not try to roll your own. There are far too many ways to get things wrong. There are some limited cases in which you might want to hand-type an XML document, but in that case, you should use an editor that can help you make sure that the XML you are typing is at least well-formed.
